Layer 0 — Geometry and Meshing
Gmsh — reuse as-is. The Python API is mature, it handles boolean geometry operations (critical for defining implant boxes, gate stacks, isolation regions), supports 2D and 3D unstructured tetrahedral meshes, and outputs to VTK natively. The gmsh.model.occ kernel does the same boolean solid geometry that Sentaurus Structure Editor does graphically. You want to wrap this, not replace it.
meshwell — reuse as-is for 2D. Developed at Google for silicon photonics pn-junction simulation with DEVSIM, it greatly facilitates the definition of meshes over freeform planar geometries characteristic of emerging integration platforms, and is used conjointly with DEVSIM for semiconductor simulations. It's your proof that this exact integration path works; you're extending it from photonics to general CMOS. Illinois
Gap: semiconductor geometry DSL. Nothing in the open source world lets you write add_implant("n-well", species="P", dose=3e12, energy=300e3) and have it automatically create the implant zone, update the mesh refinement strategy, and pass a tagged region to the doping solver. You build this.

Layer 1 — Process: Topography
ViennaPS — reuse as-is, this is your strongest reuse candidate for process. It is an open-source software framework for simulating fabrication processes in semiconductor manufacturing, with a focus on topography evolution during etching and deposition. It uses a high-performance level-set method with a hierarchical run-length encoding data structure for efficient and fast geometry representation and evolution. ViennaPS supports both analytical and physics-based process models, including Monte Carlo ray tracing for flux calculation at the feature scale. It has a Python API, is licensed under the MIT License, and is under heavy development with an active release cadence — v4.0.0 just came out. It even has optional GPU acceleration via CUDA runtime loading. There is also ViennaFit, a companion Python package for fitting ViennaPS models to experimental data — relevant for your calibration pipeline. ScienceDirect + 2
ViennaRay — reuse as-is. A ray-tracing library for topography simulations based on Intel's Embree ray tracing kernel, designed to provide efficient and high-performance ray tracing. This handles directional RIE etch profiles correctly. GitHub
FLOOXS — reuse partially, primarily for oxidation physics reference. FLOOXS covers 2D process simulation including oxide deposition, sputtering, metal oxidation, and process steps for digital superconducting devices. But the Tcl interface and stale build system make it hard to integrate. Use it as a physics reference implementation for Deal-Grove, not as a dependency. Ufl
Gap: oxidation moving boundary. The actual moving-boundary problem (SiO2 consumes Si, volume expansion ~2.2×, stress-dependent growth rate) is not properly solved in any open source tool in an integrable way. ViennaPS handles geometry evolution but doesn't know about Si→SiO2 chemistry and volumetric expansion. You'll implement Deal-Grove growth rates and couple them to ViennaPS's level-set advection.

Layer 2 — Process: Doping
Iradina — reuse with work. Iradina is truly open-source, written in C, easy to read, and very fast — about two orders of magnitude faster than SRIM. It handles BCA binary collision physics correctly for implant profiles. The limitation is it's not designed as a library — you'd call it as a subprocess or wrap its C code. For a v1 pragmatic approach, pre-compute Pearson IV distributions from Iradina for your target species/energy space and store as lookup tables, exactly as commercial tools do internally. Epj-n
Important correction from prior conversation: IM3D claims to be open-source but its transport routines are included in an executable archive which must be downloaded separately — it is not truly open-source. Cross it off the list. Epj-n
FiPy (NIST) — reuse with work for diffusion. The FiPy framework includes terms for transient diffusion, convection and standard sources, enabling the solution of arbitrary combinations of coupled elliptic, hyperbolic and parabolic PDEs. It's FVM-based (compatible with your mesh format), Python-native, NIST-maintained, and has been used for electrodeposition moving-boundary problems directly analogous to yours. Plugging Fair model diffusivities for B/P/As in Si is straightforward. GitHub
Gap: implant parameter database. No open source project has a curated table of Pearson IV moments (mean range Rp, straggle ΔRp, skewness, kurtosis) for standard implant species × energies × substrates. This is a YAML/JSON file you build and maintain. Tasch et al. (1989) and the SIMS-calibrated tables in the NIST ASTAR database give you the starting data.
Gap: TED/OED. Transient Enhanced Diffusion (from implant damage supersaturation) and Oxidation Enhanced Diffusion are where process simulation gets hard. These require a point defect kinetics model (interstitial concentration, vacancy concentration, as extra PDE fields coupled to dopant diffusion). FiPy can solve the PDE system if you write the equations; the physics model itself is what you must implement.

Layer 3 — Device Simulation
DEVSIM — reuse as primary device solver. It uses finite volume methods for TCAD device simulation with user-specified PDEs, supports importing 3D meshes, and is licensed under Apache 2.0. The Python interface allows you to define custom PDEs — mobility models, recombination, contact physics — as symbolic expressions. The GDSFactory integration already demonstrates that DEVSIM solves equations on unstructured meshes, interfaces with Gmsh for 2D and 3D meshes, and can run semiconductor simulations from the gdsfactory layout. This is your confirmed integration path. GitHubGdsfactory
Charon (Sandia) — understand but don't build on. It is written in C++ and relies on Trilinos for nonlinear/linear solvers, finite-element/finite-volume libraries, and I/O, and supports simulation of extremely large problems on massively parallel computing systems. The problem is the build complexity — it basically requires a Sandia-internal environment to build, and contributing to it means working within Sandia's institutional flow. Study its physics models and solver strategies, especially its EFFPG discretization for robustness, but don't make it a dependency. GitHub
NanoTCAD ViDES — reuse for NEGF extension later. Critical correction: NanoTCAD ViDES is an open-source software suite that enables the self-consistent solution of electrostatics through the Poisson equation and electronic transport in both the quantum elastic regime via NEGF formalism and the semi-classical dissipative regime via drift-diffusion equations, in nanoscopic or mesoscopic devices of any dimension. A 2023 paper confirms it's actively maintained and handles 2D materials, heterostructures, and printed electronics. This is your NEGF backend for sub-20nm regime simulations, not your default DD solver. PubMed Central
ViennaSHE — niche. Useful if you need full Boltzmann transport for hot carrier injection modeling, but this is a Phase 3 concern at earliest.

Layer 4 — Materials and Calibration
openbandparams — limited reuse. It provides band parameters for III-V zinc-blende binaries, ternaries, and quaternaries including lattice constants, thermal expansion, bandgap, and alloy composition dependencies. Last updated 2015. Useful for III-V device extensions but has no Si, SiO2, Si3N4, or poly-Si — the materials you need first. GitHub
SKY130 / IHP SG13G2 PDKs — use as calibration targets. Both have open process specs, published device I-V characteristics, and active communities. IHP SG13G2 is particularly interesting because it includes SiGe:C HBTs where your ECoG background would give you domain insight.
Gap: Si/SiO2/SiN material parameter database. This is entirely absent in open source. You need: Si electron/hole mobility (Klaassen model parameters), SiO2 permittivity and fixed oxide charge, Si3N4 parameters, SRH recombination parameters (τ_n, τ_p, trap energy), Auger coefficients, bandgap narrowing at high doping (Slotboom model), interface trap density Dit as a function of process conditions. All with literature citations and uncertainty ranges. Build this as a structured YAML repository from day one.

Layer 5 — Integration and Pipeline
GDSFactory — partial reuse. It provides a unified Python-driven workflow spanning layout design, device simulation through FDTD and FEM tools, and TCAD via DEVSIM. It is photonics-focused but the DEVSIM integration demonstrates the exact pattern you want: Python API → Gmsh → DEVSIM → VTK output. You can either fork its TCAD integration layer or use it as a structural template for your own pipeline. GitHub
The central gap: process-to-device mesh translator. This is the piece that doesn't exist anywhere. After ViennaPS evolves the topography and FiPy computes the doping profiles on a process mesh, you need to:

Extract the final material regions from the level-set representation
Generate a new device mesh (via Gmsh) conforming to the material interfaces
Interpolate doping fields from the process mesh onto the device mesh
Tag contacts, oxide/semiconductor interfaces, and material boundaries
Pass everything to DEVSIM in its expected format
